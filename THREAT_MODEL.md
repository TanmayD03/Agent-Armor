## 1. Purpose

This document models the threats that kvlr is **designed to defend against** (threats to the *systems it protects*) as well as threats **against kvlr itself** (threats that could undermine the tool's integrity). The fund committee should note that a security tool with a weak threat model is itself a liability — kvlr is designed to be held to the highest standard.

---

## 2. System Overview

### What kvlr Does

kvlr is a Zero-Trust middleware layer that sits between an AI coding agent and a production codebase. Every code block generated by an AI passes through a 7-stage pipeline before reaching the developer's filesystem or CI/CD pipeline.

```
┌─────────────────────────────────────────────────────────────────────┐
│                         TRUST BOUNDARY                              │
│                                                                     │
│  [AI Agent / LLM]  ──►  [ kvlr Pipeline ]  ──►  [Production Code]  │
│                                 │                                   │
│                    ┌────────────┼────────────┐                      │
│                    │           │             │                      │
│              Stage 1-5     Stage 6       Stage 7                    │
│             (Detection)  (Attestation) (Shadow-Chain)               │
└─────────────────────────────────────────────────────────────────────┘
```

### Components in Scope

| Component | File | Role |
|-----------|------|------|
| **Secret Scrubber** | `agent_armor/core/secret_scrubber.py` | Regex + entropy-based secret detection and replacement |
| **AST Hardening Engine** | `agent_armor/core/ast_hardener.py` | Static analysis of Python AST for dangerous sinks |
| **DTG Engine** | `agent_armor/core/dtg_engine.py` | Data Transformation Graph — taint tracking from source → sink |
| **Slopsquatting Guard** | `agent_armor/guards/slopsquatting_guard.py` | Dependency hallucination and supply-chain attack detection |
| **Policy Engine** | `agent_armor/core/policy_engine.py` | Semantic rule enforcement (BOLA, JWT, auth, debug mode…) |
| **Attestation Engine** | `agent_armor/core/attestation.py` | SHA-256 invariant signing |
| **Shadow-Chain Manager** | `agent_armor/shadow_chain/chain_manager.py` | Append-only tamper-evident audit ledger |
| **MCP Security Proxy** | `agent_armor/mcp_proxy/interceptor.py` | Man-in-the-middle for Model Context Protocol tool calls |
| **CLI** | `cli/armor.py` | Developer-facing interface |

### Out of Scope

- The AI model itself (GPT-4, Claude, Gemini, etc.)
- The developer's IDE or editor
- The MCP server downstream of the proxy
- Network infrastructure between the developer and the AI provider

## 3. Assets to Protect

| Asset | Confidentiality | Integrity | Availability | Notes |
|-------|:-:|:-:|:-:|-------|
| Developer source code (input) | HIGH | HIGH | MEDIUM | Could contain business logic, IP |
| Secrets in source code | CRITICAL | — | — | API keys, DB passwords, private keys |
| Shadow-chain ledger | — | CRITICAL | HIGH | Tampered chain → false attestations |
| PyPI/npm registry cache | — | HIGH | MEDIUM | Poisoned cache → bypassed dep checks |
| Attestation signatures | — | CRITICAL | — | Forged sig → tampered code looks attested |
| kvlr itself (the tool binary/package) | HIGH | CRITICAL | HIGH | Compromised tool → all protections bypassed |
| CI/CD pipeline decisions | — | CRITICAL | — | Wrong APPROVED/BLOCKED → security bypass |

## 4. Threat Actors

| Actor | Motivation | Capability | Likelihood |
|-------|-----------|-----------|-----------|
| **Malicious AI Training Data** | Poison AI output to include backdoors | High — indirection via model | HIGH |
| **Supply-chain attacker** | Publish look-alike packages to PyPI | Medium — requires PyPI account | HIGH |
| **Insider / Compromised Developer** | Bypass attestation to ship vulnerable code | High — direct filesystem access | MEDIUM |
| **Adversarial Prompt Engineer** | Craft prompts that produce code which evades detection | Medium — knowledge of kvlr rules required | MEDIUM |
| **Nation-state / APT** | Long-term compromise of developer toolchain | Very high — well-resourced | LOW (but HIGH impact) |
| **Automated Exploit Scanner** | Opportunistic — scan public repos for weaknesses | High — automated | MEDIUM |

## 5. STRIDE Analysis

### 5.1 Spoofing

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **S1** | Attestation Engine | Attacker forges a `@kvlr-attestation:` header with a valid-looking SHA-256 to make tampered code appear attested | `AttestationEngine.verify()` recomputes the hash from scratch and compares — forged hashes will not match | ✅ Mitigated |
| **S2** | Shadow-Chain | Attacker deletes the chain file and creates a new one with fabricated entries | `prev_hash` links entries; deleting the chain breaks `chain_hash` continuity; CI `armor verify` detects absence of chain | ✅ Mitigated |
| **S3** | Slopsquatting Guard | Attacker registers `requets` on PyPI (typosquatting) hoping an AI hallucinates it | Levenshtein distance check against `_POPULAR_PACKAGES` list flags packages with edit-distance ≤ 2 | ✅ Mitigated |
| **S4** | MCP Proxy | Attacker sends a crafted MCP payload with a spoofed `tool_name` to bypass interception | Proxy validates all tool call fields; unknown tool types are blocked by default | ✅ Mitigated |

### 5.2 Tampering

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **T1** | Shadow-Chain ledger | Developer manually edits `.kvlr/shadow-chain.json` to cover tracks after removing a security check | `chain_hash` is SHA-256 over all entries concatenated; any edit breaks the hash | ✅ Mitigated |
| **T2** | Attested source file | Developer removes `if user.is_admin:` guard after attestation | `AttestationEngine.verify()` re-derives hash from current code; mismatch → CI blocks deploy | ✅ Mitigated |
| **T3** | PyPI cache file | Attacker with local filesystem access modifies `.kvlr/pypi-cache.json` to whitelist a malicious package | Cache miss on unknown entries triggers live PyPI API re-validation; cache TTL limits stale data window | ⚠️ Partial — future: cache signing |
| **T4** | kvlr package itself | Malicious PyPI package (supply-chain attack on kvlr) | kvlr self-attests on every CI release; SLSA Level 2 provenance via GitHub Actions | ✅ Mitigated |
| **T5** | AST Hardener bypass | Unicode-normalised function names (`ｅｖａｌ` vs `eval`) evade pattern matching | **Known gap** — identifier normalisation via `unicodedata.normalize('NFKC')` planned for v1.1.0 | ⚠️ Known gap |

### 5.3 Repudiation

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **R1** | Pipeline | Developer claims they didn't approve a piece of vulnerable code | Every APPROVED/WARNED scan writes a timestamped entry to the Shadow-Chain | ✅ Mitigated |
| **R2** | CI/CD | Build system claims code was never scanned | `@kvlr-attestation:` header embedded in source file is immutable proof of scan | ✅ Mitigated |
| **R3** | Slopsquatting Guard | Developer claims they didn't knowingly import a malicious package | Package validation results stored in `ArmorReport` and optionally in `--output report.json` | ✅ Mitigated |

### 5.4 Information Disclosure

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **I1** | Secret Scrubber | Developer's API key included in code passed to kvlr is logged to disk | kvlr never writes the original code to disk; only the scrubbed version enters the shadow-chain | ✅ Mitigated |
| **I2** | Slopsquatting Guard | Package names queried against live PyPI API could leak project dependency graph | PyPI API is public; requests contain only the package name (no auth, no code) | ✅ Acceptable |
| **I3** | MCP Proxy | Tool call payloads (potentially containing partial source code) logged to stdout | Proxy does not log payloads at INFO level; DEBUG logging is opt-in and documented as sensitive | ✅ Mitigated |
| **I4** | ArmorReport | `report.to_json()` includes `original_code` — if written to disk, secrets in pre-scrub code could be exposed | Documented: users should use `--output` with appropriate file permissions; `original_code` field marked sensitive in docs | ⚠️ Documented risk |

### 5.5 Denial of Service

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **D1** | AST Hardener | Adversarially crafted Python code with deeply nested ASTs causes `ast.parse()` to consume unbounded memory | Python's `ast.parse()` has depth limits; hardener wraps parse in try/except and returns a BLOCKED result on failure | ✅ Mitigated |
| **D2** | DTG Engine | Extremely large code files cause O(n²) graph traversal | DTG engine processes files < 1MB; larger inputs are BLOCKED with a warning | ✅ Mitigated |
| **D3** | Slopsquatting Guard | PyPI API outage causes guard to block all installs | Offline fallback mode: guard operates on blocklist + typosquatting checks only; CI does not fail on network timeout | ✅ Mitigated |
| **D4** | Shadow-Chain | Very long-running project accumulates thousands of chain entries causing slow verify | Chain verification is O(n) on entries; future: periodic chain compaction planned | ⚠️ Future work |

### 5.6 Elevation of Privilege

| Threat | Component | Attack Scenario | Mitigation | Status |
|--------|-----------|----------------|-----------|--------|
| **E1** | MCP Proxy | AI agent crafts a `write_to_file` payload with path `../../etc/passwd` (path traversal) | `_FORBIDDEN_PATHS` regex blocks writes to `/etc/`, `~/.ssh/`, `~/.aws/`, `.env`, `C:\Windows\System32` | ✅ Mitigated |
| **E2** | Policy Engine | AI-generated Flask route missing `@login_required` on `/admin` endpoint | `RULE-004`: Admin Endpoint Auth rule flags any route matching `/admin` without auth decorator | ✅ Mitigated |
| **E3** | Policy Engine | AI-generated code deletes records without scoping to `user_id` (BOLA/IDOR) | `RULE-002` + `RULE-009`: Delete-without-user-id and Broken Object Level Auth rules | ✅ Mitigated |
| **E4** | Policy Engine | `debug=True` shipped to production exposes stack traces and interactive debugger | `RULE-010`: Insecure Design rule catches `debug=True` as CRITICAL | ✅ Mitigated |

## 6. Attack Trees

### 6.1 "Bypass the Attestation Check"

```
Goal: Deploy tampered code that passes kvlr verify
│
├── [A] Forge the attestation header
│   ├── Brute-force SHA-256 preimage          [INFEASIBLE — 2^256]
│   └── Steal the signing key                [N/A — kvlr uses no private key;
│                                             hash is deterministic from code+invariants]
│
├── [B] Tamper with the Shadow-Chain
│   ├── Delete chain file                    [DETECTED — CI checks chain exists]
│   ├── Edit entries                         [DETECTED — chain_hash mismatch]
│   └── Replace entire chain                 [DETECTED — prev_hash chain broken]
│
├── [C] Bypass pipeline entirely
│   ├── Commit code without running kvlr     [DETECTED — @kvlr-attestation: header absent]
│   └── Disable CI check                     [REQUIRES repo admin access — out of scope]
│
└── [D] Exploit a kvlr bug to produce wrong APPROVED result
    ├── Unicode bypass in AST Hardener       [KNOWN GAP — v1.1.0 fix planned]
    ├── Crafted code that crashes pipeline   [BLOCKED result returned on error]
    └── Race condition in chain write        [File locking — future work]
```

### 6.2 "AI Dependency Confusion Attack"

```
Goal: Get a malicious package installed via AI-generated import
│
├── [A] Hallucinated package name
│   ├── Register exact hallucinated name on PyPI  [DETECTED — existence check]
│   └── Hope developer installs without scanning  [DETECTED — SlopsquattingGuard]
│
├── [B] Typosquatting
│   ├── Edit distance ≤ 2 from popular package    [DETECTED — Levenshtein check]
│   └── Edit distance > 2 (less obvious typo)     [PARTIAL — depends on threshold]
│
├── [C] New malicious package
│   ├── Package < 7 days old                      [DETECTED — age check]
│   └── Package < 1,000 downloads                 [DETECTED — download check]
│
└── [D] Long-con supply chain attack
    ├── Maintain legitimate package for 1+ year   [HARD — requires sustained effort]
    └── Introduce malicious version later         [PARTIAL — version pinning recommended]
```

## 7. Trust Boundaries

```
┌────────────────────────────────────────────────────────────────┐
│  UNTRUSTED ZONE                                                │
│                                                                │
│  • AI-generated code (treated as fully untrusted input)        │
│  • MCP tool call payloads                                      │
│  • PyPI registry responses                                     │
│  • User-supplied filenames and paths                           │
└────────────────┬───────────────────────────────────────────────┘
                 │  ALL INPUT CROSSES THIS BOUNDARY
                 ▼
┌────────────────────────────────────────────────────────────────┐
│  kvlr PIPELINE (semi-trusted)                                  │
│                                                                │
│  • Runs entirely in user's process — no network calls          │
│    except Slopsquatting Guard → PyPI (read-only, no auth)      │
│  • Produces ArmorReport (trusted output)                       │
│  • Writes Shadow-Chain (trusted artifact)                      │
└────────────────┬───────────────────────────────────────────────┘
                 │  ONLY APPROVED/WARNED CODE CROSSES THIS BOUNDARY
                 ▼
┌────────────────────────────────────────────────────────────────┐
│  TRUSTED ZONE                                                  │
│                                                                │
│  • Developer filesystem                                        │
│  • CI/CD pipeline                                              │
│  • Production codebase                                         │
└────────────────────────────────────────────────────────────────┘
```

## 8. Known Gaps & Residual Risks

These are documented, accepted residual risks — not oversights:

| ID | Gap | Risk Level | Planned Fix |
|----|-----|-----------|-------------|
| **G1** | Unicode/homoglyph identifier bypass in AST Hardener | MEDIUM | `unicodedata.normalize('NFKC')` — v1.1.0 |
| **G2** | PyPI cache file is unsigned — local attacker could whitelist packages | LOW | Cache signing with HMAC — v1.1.0 |
| **G3** | Shadow-Chain does not use file locking — concurrent writes could corrupt chain | LOW | `fcntl`/`msvcrt` file lock — v1.1.0 |
| **G4** | ArmorReport `original_code` field includes pre-scrub secrets if serialised | LOW | Documented; `--output` flag warns about sensitive content |
| **G5** | JavaScript/TypeScript not yet supported — AI-generated JS code is not scanned | HIGH (scope) | JS/TS engine — v1.2.0 roadmap |
| **G6** | Long-established malicious packages (> 1 year old, > 1k downloads) evade age/download heuristics | MEDIUM | Integration with OSV.dev vulnerability database — v1.1.0 |

## 9. Security Properties Guaranteed by kvlr

The following are **hard guarantees** — properties that hold regardless of the content of the AI-generated code:

1. **No code execution**: kvlr uses `ast.parse()` only. It never `eval()`s or `exec()`s any input code.
2. **No network exfiltration**: The only outbound network call is a read-only GET to `https://pypi.org/pypi/<name>/json`. No source code is transmitted.
3. **Air-gap compatible**: With `validate_packages=False`, kvlr operates with zero network calls — safe for classified/air-gapped environments.
4. **Deterministic attestation**: The same code + the same invariants will always produce the same SHA-256 signature. No randomness, no timestamps in the hash input.
5. **Fail-closed**: Any unhandled exception in the pipeline returns `status=BLOCKED`, not `APPROVED`. The tool fails safe.
6. **No global state mutation**: Each `AgentArmor().process()` call is stateless. Parallel calls are safe.
